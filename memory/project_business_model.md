El fundador tiene convicción open source. Cualquier monetización futura debe ser compatible.

**Modelos analizados:**

- **Open Core** — núcleo open, features enterprise privadas (GitLab, Metabase). Riesgo: la comunidad percibe que lo bueno está detrás del muro.
- **AGPL + licencia comercial** — código público, empresas pagan por excepción AGPL (MongoDB, MariaDB). Técnicamente puro pero complejo de gestionar.
- **Open Source + SaaS** — código 100% público, se cobra por el servicio gestionado (Plausible, Ghost). La propuesta de valor es conveniencia, no features exclusivas.
- **Fair Source / BSL** — código público, libre para uso personal, open source completo tras N años (Sentry, HashiCorp).

**Opciones preferidas para Scholio:**

1. **Open Source + SaaS** — código completamente público, Scholio Cloud como servicio gestionado para quien no quiera self-hostear. Muy alineado con valores académicos de transparencia.
2. **Open Core** — si hay features que solo tienen sentido a escala institucional (SSO, auditoría, SLA).

**Segmento institucional:**
Universidades e instituciones pagarían por hosting propio con soporte — compatible con open source y ticket medio alto.

**How to apply:** Al decidir qué features construir, preguntarse si pertenecen al núcleo open o al tier institucional. No meter detrás de paywall nada que sea core para el flujo académico individual.
